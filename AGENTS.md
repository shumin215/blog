# Codex Agent Guide

This repository is Shumin's World, a personal technical blog built with Astro 6,
MDX, Tailwind CSS v4, Pagefind search, Giscus comments, and Cloudflare deployment.

Use this file as the first operational reference when working as a Codex agent.
`CLAUDE.md`, `.claude/agents/*.md`, and `README.md` contain overlapping background
context, but this file is the concise source for coding workflow and
project-specific guardrails.

## Project Shape

- `src/pages/`: route pages. Important routes include `index.astro`, `search.astro`,
  `personal.astro`, `playground.astro`, `rss.xml.ts`, `posts/[slug].astro`, and
  `tags/[tag].astro`.
- `src/components/`: reusable Astro components such as `Header`, `Footer`,
  `PostCard`, `Callout`, `ImageCaption`, and `Comments`.
- `src/layouts/BaseLayout.astro`: common HTML shell, SEO metadata, OG/Twitter tags,
  and dark-mode initialization.
- `src/content/posts/`: MDX blog posts. The file name becomes the post slug.
- `src/content.config.ts`: Astro content collection schema.
- `src/styles/global.css`: Tailwind CSS setup, typography, dark mode, and code block
  styling.
- `src/utils/`: small shared helpers such as reading-time and URL helpers.
- `.github/workflows/`: CI/CD for main deployment and PR previews.
- `public/`: static assets such as favicon and robots.txt.
- `dist/`: generated build output. Do not edit by hand.

## Commands

Use Node.js 22 or newer.

```bash
npm install
npm run dev
npm run build
npm run preview
npm run lint
npm run format
```

- `npm run dev` starts the Astro dev server, usually at `http://localhost:4321`.
- `npm run build` runs `astro build` and then `pagefind --site dist`.
- Use `npm run preview` after a production build when checking Pagefind search.
- Run `npm run lint` after code changes when practical.
- Run `npm run build` after changes that affect routes, content collections, MDX,
  RSS, sitemap, Pagefind, or deployment behavior.

## Coding Conventions

- Prefer the existing Astro component style and local helpers before introducing
  new abstractions.
- Keep TypeScript strict-friendly. Avoid `any` unless there is a concrete reason.
- Use PascalCase for component files, for example `PostCard.astro`.
- Use kebab-case for normal route file names when adding new pages.
- Keep formatting compatible with `prettier.config.mjs`: 2 spaces, semicolons,
  single quotes, trailing commas where valid, and 100 character print width.
- Tailwind classes should follow the existing mobile-first style.
- Avoid editing generated output such as `dist/` or package manager artifacts unless
  the task specifically requires dependency changes.

## Content Rules

Posts live in `src/content/posts/` and may be `.md` or `.mdx`.

Required frontmatter:

```yaml
---
title: "Post title"
date: 2026-03-15
description: "Short SEO-friendly description"
tags: ["tag1", "tag2"]
---
```

Optional frontmatter:

```yaml
category: "Programming"
draft: false
cover: "/images/posts/cover.png"
```

- `category` must be one of `AI`, `Computer Architecture`, `Programming`,
  `Digital Logic`, `NAS`, or `Series`.
- `draft: true` posts are treated as drafts by the site logic.
- MDX posts may use:

```mdx
<Callout type="info">Information</Callout>
<Callout type="tip">Tip</Callout>
<Callout type="warn">Warning</Callout>
<Callout type="danger">Danger</Callout>

<ImageCaption src="/images/example.png" alt="Description" caption="Caption" />
```

## Feature Notes

- Search is powered by Pagefind and depends on build output in `dist/pagefind`.
- Comments use Giscus in `src/components/Comments.astro`.
- The header owns navigation, search modal behavior, keyboard shortcut handling,
  and dark-mode toggle behavior.
- Post detail pages include table of contents, reading time, previous/next post
  navigation, and comments.
- RSS is generated by `src/pages/rss.xml.ts`.
- Sitemap support is configured through `@astrojs/sitemap` in `astro.config.mjs`.

## WordPress Migration Skill

Use the repo-local Codex skill at
`.codex/skills/wordpress-migration/SKILL.md` when a task mentions WordPress,
Synology NAS, migration, legacy posts, converting WordPress HTML to Astro MDX,
or creating a migration PR.

The older `.claude/agents/` directory documents the same workflow as three
Claude sub-agents. Treat those files as source references, but prefer the Codex
skill for operational guidance.

- Phase 1 maps to `.claude/agents/wp-reader.md`: read one WordPress post from the
  Synology NAS and write `/tmp/wp-migration/post.json`.
- Phase 2 maps to `.claude/agents/content-processor.md`: convert the JSON and
  copied images into Astro MDX plus public image assets.
- Phase 3 maps to `.claude/agents/migration-pr.md`: apply files to the repo,
  validate the build, create `feat/migrate-{slug}`, commit, push, and open a PR.

When doing this workflow as Codex:

- Ask for a target post ID, slug, or title if the user has not specified one.
- Before migrating, compare WordPress slugs with existing `.md` and `.mdx` files
  in `src/content/posts/` and warn before overwriting an already migrated post.
- Do not print database passwords, tokens, SSH private key contents, or other
  secrets; redact secrets from command errors before summarizing them.
- Copy only images referenced by the target post, not the full WordPress uploads
  directory.
- Keep temporary migration artifacts in `/tmp/wp-migration/` unless the user asks
  to clean them up.
- Validate generated MDX frontmatter against `src/content.config.ts`.
- Run `npm run build` before creating a migration commit or PR.
- If creating branches, pushing, or opening PRs, confirm the intended branch name,
  stage only exact migration files, and avoid disturbing unrelated local changes.

## Deployment

- The production workflow runs on pushes to `main`.
- `.github/workflows/deploy.yml` installs with `npm ci`, builds, then deploys via
  Cloudflare Wrangler.
- `.github/workflows/preview.yml` builds pull requests and deploys Cloudflare Pages
  previews.
- `wrangler.toml` serves static assets from `./dist`.
- Required GitHub secrets are `CLOUDFLARE_API_TOKEN` and
  `CLOUDFLARE_ACCOUNT_ID`.

## Agent Workflow

- Start by checking `git status --short` and avoid overwriting user changes.
- Read nearby files before editing; this project is small enough that local context
  usually beats guessing.
- Keep changes scoped to the requested behavior.
- For UI changes, preserve the existing blog look: quiet, readable, responsive,
  and typography-first.
- For content-only changes, validate frontmatter against `src/content.config.ts`.
- For behavior or layout changes, prefer `npm run lint` plus `npm run build` when
  time allows.
- Mention any commands that could not be run, especially build or lint.
